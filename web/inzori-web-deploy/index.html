What inZOR-ND is claiming

  • It is a discovery engine, not a single algorithm. The core engine stays stable while the domain evaluator changes.
  • It is strongest at the hard edge of workflows where fixed rules, manual choices, or standard heuristics become fragile.
  • It is evidence-driven. Claims are tied to reproducible test pages, not to a promise that one method wins everywhere.
How to read the project

inZOR-ND is the scientific face of the work. inZORi is the underlying evolutionary engine. The important question is not whether one fixed formula is universal, but whether the same engine can repeatedly discover useful rules when the evaluator is real and the search space is difficult.

1. What inZOR-ND is

inZOR-ND is a way to turn hard scientific workflows into evolutionary worlds. A candidate can be an orbital selection rule, a warm-start policy, a circuit structure, a plasma indicator, or another compact behaviour. The domain supplies the evaluator; the engine supplies variation, selection, persistence, and pressure toward useful structure.

The result is not a black-box prediction. The goal is usually a rule, policy, configuration, or candidate set that can be inspected and re-tested under the same protocol.

Environment

The domain defines the world

Each problem brings its own simulator, solver, benchmark, constraints, or physical evaluator. inZOR-ND does not flatten every domain into the same reward formula.

Population

Candidate behaviours compete

A population explores many alternatives. Weak candidates disappear; robust candidates survive, mutate, and are tested again.

Discovery

Useful rules emerge from pressure

The strongest outcomes are compact behaviours found by repeated evaluation, not manually tuned narratives after the fact.

2. Where it is strongest

Hard search spaces

Useful when the number of possible configurations is too large for manual scanning, but each candidate can still be evaluated.

Fragile heuristics

Useful where standard rules work in easy regimes but degrade near transitions, failures, noisy states, or hard physical boundaries.

Implicit objectives

Useful when success is better measured by a real workflow than by a closed-form target: convergence, feasibility, hardware result, or physical validity.

3. How the engine is used

  1. Define the world. Choose the real evaluator: molecule, solver, circuit, physical data stream, or grid workflow.
  2. Encode candidate behaviours. Keep the representation small enough to inspect, but expressive enough to evolve.
  3. Run selection under the real protocol. Candidates are judged by the workflow they must survive, not by a synthetic proxy alone.
  4. Freeze and validate. Promising results are separated from the search run and re-tested as fixed artefacts on held-out or independent settings.

This page stays at principle level. The detailed evidence lives in the research test pages, where each claim is tied to a specific protocol and result.

4. Good fit / bad fit

✓ Strong at

  • Combinatorial discovery problems where the useful structure is hard to write down in advance
  • Implicit or evaluator-defined objectives where success is best measured by simulation, convergence, or physical validity
  • Cross-domain reuse of the same search engine with different evaluators
  • Scientific problems where fixed heuristics degrade at the hard edge of the workflow

✗ Not the right fit for

  • Simple one-shot inference tasks where a direct model lookup is the real need
  • Problems with a single trivial optimum where evolutionary search adds no value
  • Ultra-low-latency settings where there is no room for iterative search
  • Claims of universal superiority on every benchmark class; that is not the project message
